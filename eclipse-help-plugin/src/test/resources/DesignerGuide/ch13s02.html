Web Service Configuration Editor

With the help of this master-details editor you can build up your own web service configuration tree: Add, remove, and change both folders and WS entries.

Overview

This chapter describes how to use the web service configuration editor. At the end you will be able to create, delete, and change your web service configurations.

Web service configuration tree editor

Web service configurations are displayed in a tree hierarchy. The tree editor helps to create new, and delete existing tree elements (web service folders, and web service configuration entries).

Figure 13.1. Web service tree editor part


WS tree editor is highlighted with a red rectangle on the screenshot above. In the editor the folder elements are symbolized with a folder icon. The ones, which are not empty, are marked with a small collapsible node (just like the "xhrm finance" on the screenshot). The web service configuration elements are symbolized with a globe icon. There are the controller icons on the right side of the tree editor within the green rectangle. These buttons offer following functions:

Create new configuration Folder

Create a new configuration folder. Only for structuring your services

Create new Web Service Configuration

Create a new Web Service Configuration. The system

Add environment

Add an environment configuration for the selected Web Service. Select one or more environments for the environment list

Remove

Remove the current selection (configuration entry, WS folder or event a WS Subtree

Warning

Mind: entries you want do erase, might still be in use. It is encouraged to remove generated client classes as well with your service entry, but take care: some of these classes might be referred by other WS entries as well.

Web service folder/entry editor

Details of currently selected web service configuration element will be displayed on the right hand side. This editor helps to change details of an existing element (web service folder, or web service configuration entry).

Figure 13.2. Web service configuration details part


The web service entry editor part is highlighted with a red rectangle on the screenshot above. It helps to change attributes of an existing configuration entry (or WS folder).

The following attributes are available (from above):

Name

The attribute specifies the displayed name of a web service configuration. The name is not used as identifier, so that feel free to rename existing entries.

This is the only attribute of web service folders.

Description

Description attribute is for own textual descriptions. Not used anywhere in ivy, but you are encouraged to fill it out. It helps years later to understand the exact purpose of WS configuration and the way it should be accessed.

WSDL URL

Service details and classes will be generated using the WSDL specified here. Please use protocol prefix like:

http://myserver.ch/hello.wsdl

file://c:\temp\myWis.wsdl

Use session handling

You can chose session handling mode for current configuration. There are 3 modes available now:

  • NO there will be no session handling when you use this configuration

  • WSELEMENT invoking the same service from the same "WS step" process entry existing sessions will be reused

  • APPLICATION Within ivy applications whenever you invoke the same service existing session will be reused

Use auth.

Use authentication: When this check box is set, authentication is used to access the web service.

Auth type

If the "Use auth." is checked, then here you can choose authentication type to use. Currently only simple authentication is supported. Of course when you uncheck the use authentication setting this selection becomes disabled

User name

When authentication is used, this user name will be applied to get access to this web service. You can use IvyScript in this field.

Tip

When you specify authentication in "WS step" process element, these settings (User name and Password) will be overridden. Use Scripts like in.user carefully, since you might use this WS entry in multiple processes with different data types.

Password

When authentication is used, the user name above and this password will be applied to get access to web service. IvyScript is also interpreted in this field.

Generate WS classes

When you specified the necessary fields: WSDL address, authentication details, and Axis version (1.x or 2.0) then you have to read the WSDL and generate client side classes to be able to access the web service (e.g. use its data types, read its operations, etc.). These generated files will be pre-compiled to be tested, and packed in jar file together. These jar files are located in the lib/ws_server folder of ivy project.

Note: when you change the WSDL reference or the WSDL itself you always have to regenerate service classes.

Use Axis 2.x

If you uncheck this option, the generation will be done using Axis 1.x otherwise using Axis 2.x. Unless you have some special reasons to use the older Axis implementation, please select Axis 2.x (faster, less memory usage, etc.)

Port types

This is a combo box. This choice becomes available first when you have already generated client side classes. (see: Generate WS classes)

The content of this box comes from the WSDL specified above, and being interpreted by Axis. So that your web service provider/architect can give more information about.

A port type is something like a group of operations according to some technical or business criteria. It has got nothing to do with TCP/IP, UDP or firewalls. Port types quite an often reflect only SOAP versions used to access the same operations. In such case you normally see your port names ending with SOAP12, SOAP13, etc. If so, select the latest (biggest number)

Def. endpoint*

A URL is expected to be specified here. This is the URL where the current web service is located, and going to be accessed from. After client generation you might see URL-s here, because they can be specified in WSDL. If not, you have to set it up on your own. Value of this field depends on the "Port type" selection, so that when you select another "Port type" you might see another URL in this field (if you can not specify this field, ask your web service provider).

Fallback servers

A list of servers can be specified here. They are used as fallback servers if any error happens during web service request. First, the server at default endpoint will be requested. Servers on the list are queried one by one until a successful web service access. You find error messages in runtime log when some endpoint invocations fail. If a service invocation is successful then the process continues as normal. This list is optional. When some servers are specified, then they are used, when not, then only the server at default endpoint will be accessed. If not even the default endpoint is specified then an exception is raised. (process continues at exception handler element if specified)

Environments

In the configuration editor you can define different Environments. The environments can be added to existing web service configurations. Depending on the active environment the associated environment configuration will used. If no environment configuration for the web service is available, the default configuration will be used. You can now define one web service with environment specific connection properties. For example it is possible to define a Web Service for your development, test and production environment.

If a new web service configuration is added to the container, the system automatically adds empty environment configurations for each environment defined in the project.

Note

The root web service acts as the default configuration and must always be set

Tip

You don't must define environment configurations for every web service. If the web service configuration is the same for all environments, only the default configuration is necessary.
